# Story 2.1: Establish WebSocket Connection Automatically

## Status: Draft

## Story

- **As a** developer
- **I want** to automatically establish a WebSocket connection to the backend server as soon as the app is ready
- **so that** video data can be streamed for recognition without user interaction.

## Acceptance Criteria (ACs)

1. A `WebSocketService` is created to encapsulate all connection logic.
2. The service is registered with `get_it` for global access.
3. After the camera is successfully initialized (from Story 1.2), the app automatically attempts to connect to the backend WebSocket URL.
4. Connection status changes (e.g., "Connecting...", "Connected", "Disconnected", "Error") are logged to the debug view (from Story 1.3).
5. The connection is gracefully closed when the application is paused or terminated.
6. The service includes logic for automatic reconnection with a backoff strategy if the initial connection fails or is lost.

## Tasks / Subtasks

- [ ] **Task 1: Create WebSocket Service** (AC: 1, 2)
    - [ ] Create a `WebSocketService` class.
    - [ ] Add the `web_socket_channel` dependency if not already present.
    - [ ] Encapsulate the logic for creating and managing a `WebSocketChannel`.
    - [ ] Register the service as a lazy singleton in `get_it`.
- [ ] **Task 2: Implement Connection Logic** (AC: 3, 4)
    - [ ] Create a `connect()` method in the service.
    - [ ] This method should handle the WebSocket URL (from a config file/class) and establish the connection.
    - [ ] Implement a BLoC/Cubit to manage and expose the connection state (`WebSocketState`: `Initial`, `Connecting`, `Connected`, `Disconnected`).
    - [ ] Log every state change to the `DebugLogService`.
- [ ] **Task 3: Implement Auto-Reconnection** (AC: 6)
    - [ ] In the `connect()` method, add a `try-catch` block for connection errors.
    - [ ] On failure, schedule a reconnection attempt after a delay (e.g., 2s, 4s, 8s).
    - [ ] Also, listen to the channel's `done` future to detect unexpected disconnections and trigger the reconnection logic.
- [ ] **Task 4: Integrate with App Lifecycle** (AC: 3, 5)
    - [ ] Create a main application BLoC or use an existing one to coordinate events.
    - [ ] When the Camera BLoC from Story 1.2 enters the `Streaming` state, call the `WebSocketService.connect()` method.
    - [ ] Use a `WidgetsBindingObserver` to detect app lifecycle changes (`AppLifecycleState.paused`, `AppLifecycleState.detached`).
    - [ ] Call a `disconnect()` method in the service to gracefully close the channel when the app is paused or closed.

## Dev Technical Guidance

- **Configuration**: Do not hardcode the WebSocket URL. Store it in a dedicated configuration file or class. For development, this can be a local IP.
- **State Management**: It's crucial to manage the WebSocket connection state via a BLoC. This allows the UI and other services to react to connection status changes (e.g., disable a "send" button if disconnected).
- **Graceful Shutdown**: Properly closing the WebSocket (`channel.sink.close()`) is important to free up resources on both the client and server.
- **Backoff Strategy**: For reconnection, use an exponential backoff strategy to avoid overwhelming the server with connection requests. For example: retry after 2s, then 4s, then 8s, up to a maximum delay.
