# Story 2.2: Stream Camera Frames via WebSocket

## Status: Draft

## Story

- **As a** developer
- **I want** to capture frames from the live camera feed and send them through the active WebSocket connection
- **so that** the backend can perform facial recognition.

## Acceptance Criteria (ACs)

1. When the `WebSocketService` is connected, the app begins capturing frames from the camera stream.
2. Frames are captured at a regular, configurable interval (e.g., every 250ms).
3. Each captured camera frame (`CameraImage`) is converted to a Base64 encoded string.
4. The Base64 string is sent as a message through the active WebSocket channel.
5. Frame streaming is handled efficiently to avoid performance issues or blocking the UI thread.
6. Streaming pauses when the WebSocket connection is lost and resumes automatically upon reconnection.

## Tasks / Subtasks

- [ ] **Task 1: Start Frame Streaming** (AC: 1, 6)
    - [ ] In the `CameraService`, expose the camera image stream (`controller.startImageStream()`).
    - [ ] Create a new BLoC/Cubit to manage the streaming state (`StreamingState`: `Idle`, `Streaming`, `Paused`).
    - [ ] When the WebSocket BLoC enters the `Connected` state, trigger an event in the new Streaming BLoC to begin streaming.
- [ ] **Task 2: Process and Send Frames** (AC: 2, 3, 4, 5)
    - [ ] Listen to the camera's image stream.
    - [ ] To avoid processing every single frame, use a timer or a stream sampling method (e.g., `throttle` from `rxdart`) to process frames at the desired interval.
    - [ ] For each selected frame, create an asynchronous processing task (e.g., using `compute` or an `Isolate`) to perform the heavy lifting of Base64 conversion off the main thread.
    - [ ] The processing task will take a `CameraImage` and return a `String`.
    - [ ] Send the resulting Base64 string to the `WebSocketService`.
- [ ] **Task 3: Manage Streaming Lifecycle** (AC: 6)
    - [ ] The Streaming BLoC should listen to the WebSocket BLoC.
    - [ ] If the WebSocket disconnects, the Streaming BLoC should transition to a `Paused` state, which stops the frame processing loop.
    - [ ] When the WebSocket reconnects, the Streaming BLoC should automatically transition back to `Streaming`.
    - [ ] Ensure the camera image stream (`stopImageStream`) is properly closed when the feature is paused or the app is closed.

## Dev Technical Guidance

- **Performance is Key**: Converting camera frames to Base64 is computationally expensive. **It MUST be done in a background isolate** using the `compute` function to avoid freezing the UI.
- **`CameraImage` Format**: The `CameraImage` from Flutter's camera plugin is often in YUV420 format. You will need a function to convert this to a more common format like JPG before Base64 encoding, or find a library that can handle the conversion. The `image` package can be useful here.
- **Stream Throttling**: Do not send every frame. The backend likely cannot handle 30fps from every client. Use `rxdart`'s `throttleTime` or a manual `Timer` to control the send rate. A good starting point is 4 frames per second (every 250ms).
- **Backend Contract**: Confirm the exact format the backend expects. Is it just the Base64 string, or a JSON object like `{"image": "base64string..."}`?
