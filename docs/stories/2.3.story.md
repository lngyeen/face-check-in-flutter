# Story 2.3: Process Backend Responses & Display Toasts

## Status: Draft

## Story

- **As a** user
- **I want** to see immediate feedback via a Toast notification when the backend sends a recognition result
- **so that** I know if my check-in was successful or not.

## Acceptance Criteria (ACs)

1. The app actively listens for incoming messages on the WebSocket channel.
2. Incoming messages (assumed to be JSON) are parsed successfully.
3. On a "success" message (e.g., `{"status": "success", "name": "John Doe"}`), a success Toast is displayed containing a personalized greeting.
4. On a "failure" message (e.g., `{"status": "failure"}`), a failure Toast is displayed.
5. All Toast notifications automatically disappear after a short, configurable duration (e.g., 3 seconds).
6. Malformed or unrecognized messages from the backend are gracefully handled and logged to the debug view without crashing the app.

## Tasks / Subtasks

- [ ] **Task 1: Listen for WebSocket Messages** (AC: 1)
    - [ ] In the `WebSocketService`, expose the channel's `stream` (`channel.stream`).
    - [ ] Create a new `RecognitionBloc` to handle the business logic of processing responses.
    - [ ] This BLoC should subscribe to the `WebSocketService`'s stream.
- [ ] **Task 2: Parse and Process Responses** (AC: 2, 6)
    - [ ] For each incoming message, use a `try-catch` block to parse the JSON string.
    - [ ] Create data models (e.g., `RecognitionResponse`, `SuccessResponse`, `FailureResponse`) for the expected JSON structures.
    - [ ] Map the parsed JSON to the appropriate data model.
    - [ ] If parsing fails or the message is unknown, log the error to `DebugLogService` and ignore the message.
- [ ] **Task 3: Implement Toast Notification Service** (AC: 3, 4, 5)
    - [ ] Add a Toast notification package like `fluttertoast` to the project.
    - [ ] Create a `NotificationService` to encapsulate the logic for showing different kinds of toasts (e.g., `showSuccessToast(String name)`, `showFailureToast()`).
    - [ ] Configure the toast appearance (background color, text color, duration) to match the UI design.
- [ ] **Task 4: Trigger Toasts from BLoC** (AC: 3, 4)
    - [ ] Based on the successfully parsed response, the `RecognitionBloc` will call the appropriate method on the `NotificationService`.
    - [ ] For a success response, it will pass the user's name to the notification service.
    - [ ] The UI layer does not need to be directly involved, as the `NotificationService` handles the overlay display.

## Dev Technical Guidance

- **JSON Parsing**: Use `dart:convert` for JSON decoding. For more robust solutions with code generation, consider packages like `json_serializable`.
- **Backend Contract**: The exact structure of the JSON responses is critical. The examples (`{"status": "..."}`) are assumptions. This contract must be confirmed with the backend team.
- **Decoupling**: The `WebSocketService` should only be responsible for communication. The `RecognitionBloc` is responsible for interpreting messages. The `NotificationService` is responsible for displaying feedback. This separation of concerns is key.
- **Toast Library**: `fluttertoast` is a popular choice, but other libraries like `oktoast` or building a custom overlay with `Overlay` widget are also valid options.
